1. Vision

Forklift is an open-source version control system designed from day one for serverless hosting. Git assumes a server with a filesystem and a long-running process to negotiate packs; Forklift's data model is shaped so that a repository ("warehouse") can live on AWS-native serverless primitives — S3 for storage, Lambda behind API Gateway for the control plane, ECS for long-running background work — with clients exchanging data directly with object storage rather than through a compute layer.

Two products come out of this:

  1. The Forklift CLI — a complete, fast, parallel VCS usable entirely locally (this repo).
  2. A hosting service (Forklift's GitHub/GitLab equivalent, FORK-18) — built on the serverless architecture in §4, sharing crates with the CLI via a commons split (precedent already set by rust-cli-core).

Differentiators worth protecting as the design evolves:

  • Serverless-native remote protocol — no equivalent of git's pack negotiation process; the remote is storage + thin verification functions.
  • Tamper-evident history as a first-class feature (FORK-13): signed parcels anchored at a genesis parcel make the full history — including user management — verifiable offline by any clone.
  • Collaboration data lives in the repo (FORK-9): approvals, merges, review metadata are tracked objects, portable and auditable — not locked in a hosting provider's database.
  • Parallel by default: saturate available cores (git is largely single-threaded in its core paths); Blake3 even parallelizes hashing a single large file.
  • Bounded memory: no operation should need RAM proportional to repository size.

2. Design principles

  1. Bounded memory. The staging area is sharded per directory (one inventory file per working directory) so commands load only the shards they touch. Whole-repo operations must stream shards, never accumulate them. Trade-off accepted: on repos with many small directories, per-shard overhead costs disk/wall-clock while winning on RAM — benchmark on a large real repo once stack exists.
  2. Parallelism first. Independent work items (per-directory scans, per-file hashing, compression) fan out over the shared TaskExecutor worker pool (sized to core count). New features reuse it; never a second pool.
  3. One runtime: tokio. Decided 2026-07-02. Networking (lift/lower, the service) wants async; one runtime beats two parallelism frameworks. Rule: the moment async workloads share the runtime with the file walker, filesystem calls move to spawn_blocking/tokio::fs — stack adopts this convention from the start.
  4. Versioned formats, length-prefixed strings. Every payload leads with a VLQ version code; adding a version is a match-arm. Every user-controlled string (file names, descriptions, identifiers) is length-prefixed, never terminator-delimited — file names may contain any byte. Hashes may be newline-terminated (ASCII hex can't contain one).
  5. Canonical paths. All user paths normalize exactly once at the command boundary into WarehousePath: repo-relative, /-separated on every platform, no ./.., root = empty key. The storage layer never sees anything else.
  6. Atomicity or an explicit story. Object writes are temp-file + rename (as are pallet refs and every other state file). Multi-shard staging updates tolerate interruption (partial loads kept; re-run completes). Decided & implemented 2026-07-02: the warehouse lock. Mutating commands (load, unload, stack, shift, palletize, restore, consolidate, park) hold an exclusive lock file (.forklift/lock, created with create_new, held for the whole command) so stack can never read a half-updated staging area. Read-only commands don't take it — at worst they report an in-flight state. A stale lock (hard kill) is reported with the owning PID and instructions. Done
  7. The remote is untrusted storage + thin verifiers. Anything a client uploads is unverified until the server has checked it (hash, signature, privileges). Anything a client downloads is verifiable offline (content addressing + signatures).

3. Local architecture Phase 1 done

3.1 Object store

Content-addressed, immutable objects under .forklift/objects/: Blake3 hex hash of the uncompressed loose object (VLQ header + typed payload) names the object; zstd-compressed on disk; single 2-char fan-out level (git's scheme — decided 2026-07-02 after flattening from 4 levels); atomic writes; dedupe by existence check. Object types: blob (file contents; symlinks store their target path), tree (sorted, deterministic — identical content ⇒ identical hash), compact parcel (commit: tree hash, parents, actions with operator + timestamp, description). Current format versions: loose v1, blob v1, tree v2, parcel v2 (V2026_07_02, length-prefixed). Formats documented in docs/format/, byte-accurate.

3.2 The dock (staging area)

Decision (2026-07-02): the "dock" concept (FORK-16) and the implemented per-directory inventory are the same thing — FORK-16 is re-scoped to extending the existing sharded inventory with two behaviors, not building a separate index:

  • Repopulate on shift: switching pallets rewrites the inventory shards from the target pallet's head parcel tree.
  • Compare on stack: stacking diffs the inventory against the pallet head's tree to determine changed files, streaming shard by shard.

Layout: .forklift/inventory/inv_/inv_<component>/…/data (every path component prefixed — collision-proof by construction), plus a metadata file listing tracked directories. Entries carry ctime/mtime/device/inode/size/hash for metadata-based change detection. Optional cosmetic decision: rename "inventory" → "dock" in code/terminology for warehouse-theme consistency Needs design.

3.2.1 Change detection & moves (decided 2026-07-02) In progress

Reconciliation is a per-directory merge-join — neither "iterate the index and stat files" nor "walk files and look up the index" globally. Each directory (= one shard = one parallel task) loads its shard, reads the directory once, and reconciles the two sets: present in both → stat, and if size/mtime/ctime/inode match the entry it is unchanged without reading the file (the stat-cache fast path the stored fields exist for; on mismatch, hash decides — editors that save via write-new-then-rename change the inode on modify, so inode mismatch alone never means "different file"); present only on disk → Added/untracked; present only in the shard → Deleted. The task writes the shard back and emits events (Added/Modified/Deleted). Deleted directories: pre-populate dirty_inventory_paths from the metadata file, each visited directory removes itself (already implemented), and the leftover set after the walk is the removed directories. status, load, and stack all consume the same event stream; shift/checkout runs the same reconciliation in reverse (target tree vs shard vs worktree; conflicts = local edits vs incoming changes, using the existing conflict states).

Status of the shared event stream (decided 2026-07-03): the per-file core of the merge-join is now one shared function — inventory_utils::classify_file_against_entry (stat cache with racily-clean protection first, read-and-hash on a miss) returns a fact-only verdict (UnchangedByStat / UnchangedByHash / Modified, the latter two carrying the fresh entry and the unstored blob object). load and the unstaged stocktake walk both classify with it, so their verdicts cannot drift; policy stays with the caller (load stores the blob and persists — including on the unchanged-by-hash path, which keeps re-load self-healing for missing blobs; the read-only walk drops the object and never writes, so it still needs no warehouse lock). The full event stream — one walk feeding load/stack/stocktake — is deferred: the traversal shapes genuinely differ (load descends into untracked directories, status does not; stack visits the union with head subtrees, bottom-up), and the efficiency wins would make read-only commands write the stat cache, breaking the lock rule. Revisit when a concrete consumer arrives: --json/professional mode (FORK-26), streaming progress for huge warehouses, or the Phase 3 serverless head — whichever lands first should shape the stream. The two directory-level policies deliberately differ and stay: a corrupt shard makes mutating load self-heal (rebuild the directory), while read-only stocktake surfaces it as an error.

Moves are never detected during the parallel walk — cross-directory moves are invisible to per-directory tasks by construction, and the formats don't need them: a moved file has the same content → same Blake3 → the blob is already deduped, and parcels stay move-agnostic (git's lesson: rename detection at display time beat rename recording at commit time). Detection is a cheap post-pass: join Deleted×Added events by (device, inode) first (same-filesystem renames preserve the inode; the fields are already stored), falling back to blob-hash equality. O(changeset) time and memory — accumulating events is bounded by the changeset, not the repo, so the streaming rule holds.

Build order: finish dirty-directory detection Done → stat-cache fast path (makes load incremental) Done → add a Deleted inventory state so unload stages removals instead of erasing entries Done → move-detection post-pass. The "racily clean" case (file modified within the same mtime tick as the shard write) is already handled: entries are only trusted when their mtime is strictly older than their shard's — same-tick entries are rehashed. A side effect of dirty detection worth knowing: directories newly added to .forkliftignore are treated as deleted on the next load — their entries become staged removals, like any deleted file.

Deleted-state semantics (implemented 2026-07-02): unload marks entries Deleted instead of erasing them (shards and metadata entries are kept — they are the record of the staged removal); the working directory is never touched. During load, old entries whose file is gone from disk (deleted, renamed, newly ignored, or replaced by a directory) are carried over as Deleted — the "present only in the shard" half of the merge-join, materialized into the shard. Directories that disappeared entirely (the leftover dirty set) get all their entries marked Deleted; only shards whose files are gone from the inventory folder itself are dropped from the metadata. Loading a file that is present on disk always (re-)stages it as Normal — the same way git add re-stages a file after git rm --cached. stack is the cleanup point: stacking a parcel will consume Deleted entries and prune empty shards.

3.3 Refs and history Implemented 2026-07-02

Pallets (branches, FORK-17) are mutable pointers to head parcels: files under .forklift/pallets/<name> (names may contain /, mapped to subfolders), containing the head parcel hash, updated atomically via temp+rename. The current pallet (HEAD equivalent) is the name in .forklift/pallet; a current pallet with no ref file is unborn — born by the first stack. prepare sets it to main. Byte-accurate spec: docs/format/PALLET_REFS.md. The parcel graph (parents list) gives history; multiple parents = merges (created by consolidate).

Two more state files live beside the refs, both temp+rename and both consumed by the command that completes them: .forklift/consolidation (a merge in progress: the head being merged in becomes the second parent of the next stack) and .forklift/parked (the stack of parked parcels, newest last).

Tracked metadata (FORK-15) lives under a reserved tree path (.forklift/tracked/… namespace inside the parcel tree), stored as normal blobs/trees so it inherits versioning, signing, and transport for free — this is the substrate for users (FORK-14), keys (FORK-12), privileges (FORK-10), and post-metadata (FORK-9). Implemented 2026-07-03 on the reserved office pallet with synthesized full-state parcels; see the Phase 2 roadmap rows and docs/format/TRACKED_METADATA.md. Revision arguments (2026-07-03): everywhere a point in history is named — palletize <name> [revision], diff <a> <b>, history [revision] — a revision is a pallet name or a parcel hash (unique prefix, ≥ 4 hex chars; a pallet name wins over a hash-looking string). Shifting to a bare parcel (a detached dock) is deliberately not offered: palletize-at-revision covers the use case without a detached-head state.

Decision (2026-07-05) Shipped 2026-07-05 — a meta-pallet namespace, not reserved names. Before this, the office was a real pallet reserved by its literal name office (ensure_not_office blocks it from palletize/shift/consolidate/franchise; the server special-cases if name == OFFICE_PALLET_NAME). That does not scale: more meta pallets are coming — FORK-9 post-metadata (a dedicated metadata pallet, §6.5) and §7.2 provenance — and reserving one name (plus one == "…" check) per meta pallet accretes special-cases. Decision: differentiate meta pallets from user pallets structurally, by namespace, rather than by reserved names.
  • Keep them pallets. The office is a pallet on purpose — it inherits hashing, signing and lift/lower transport for free. Meta pallets stay pallets; only their namespace changes. (Making the office "not a pallet" would throw the free transport away — rejected.)
  • Separate storage namespace. User pallets stay .forklift/pallets/<name>; meta pallets move to their own namespace (e.g. .forklift/meta/<name>). User pallet operations only ever look in the user namespace, so no user pallet name is reserved anymore — office becomes a legal user pallet name again. The shape mirrors git's refs/heads/* vs refs/notes/*: one ref space, two namespaces.
  • Addressing (reads). A bare name always resolves to a user pallet; a meta pallet is reached with a qualifier in the revision parser — @<name> (e.g. history @office, audit @office). Qualifier @ — shipped 2026-07-05 (chosen because user names can never start with @, so it is unambiguous).
  • Wire. Meta pallets travel under a distinguishable path (e.g. /v1/meta/<name> or a qualified pallet path) so the server recognizes the namespace rather than the literal OFFICE_PALLET_NAME — this is exactly what lets FORK-9/§7.2 add meta pallets with no new per-name special-cases.
  • palletize (list) shows user pallets by default, meta pallets under a separate heading (or behind --all).
  • The content guard (ensure_not_metadata_tree, which refuses materializing/merging any tree carrying the .forklift/tracked/… namespace) stays as defense-in-depth regardless.
Why now: do it before FORK-9 adds the second meta pallet — one meta pallet to migrate, not three. Pre-release + no-legacy makes the migration itself free (no old warehouses to preserve); the cost is surface, not compatibility. Touch points: core (OFFICE_PALLET_NAME → a meta namespace), the CLI guards, audit, the lift/lower flow (push_local_trust lifts the office first), the server's == OFFICE_PALLET_NAME checks (chain/privilege verification, re-genesis adoption), and REMOTE_PROTOCOL.md.

3.3.1 Consolidation (merge) semantics (implemented 2026-07-02)

consolidate <pallet> merges the target's head into the current pallet: no-op when already contained, fast-forward when possible, otherwise a three-way merge against the common ancestor (BFS over the parcel graph). The tree walk skips subtrees identical between the sides or unchanged on theirs. File-level merging uses an exact, whitespace-preserving LCS diff3 — deliberately independent from the display diff (recorded as a rule: display heuristics must never decide a merge; both engines now compare exact bytes, but they remain separate concerns). Clean merges stack a two-parent parcel immediately; conflicts land in the working directory with diff3 markers (including the base version), the entries enter a conflict state (stocktake reports them, stack refuses until they are resolved and loaded). Binary and type-conflicting files fall back to whole-file conflicts. Untracked files are never overwritten — collisions abort before anything is touched. Open item: conflict entries currently use a single conflict state per name; per-parent conflict stages (the three existing parent-conflict states) need a multi-entry-per-name design when three-parent merges arrive.

3.4 Crate architecture (decided 2026-07-02) Split 2026-07-03

Forklift is modular: a core library holds all base logic (objects, inventory, refs, diff, protocol types), and thin heads adapt it to execution environments:

  • forklift-core — the base logic. Rule, enforced: core logic never prints, never exits, never assumes a terminal — it returns data and errors; heads own all presentation (last violation, prepare_warehouse, fixed with the split: it returns the created-pieces notes now).
  • forklift (client CLI) — the command surface, on clap since 2026-07-03 (replacing rust-cli-core, which is retired from this repo): every command and subcommand has a generated help entry with its flags, arguments and visible aliases; typos get suggestions; forklift help <command> [subcommand] and the h alias are preserved via a thin custom help command. The binary is named forklift (was lonic-forklift).
  • forklift-server (server head) — a self-hostable server, implemented 2026-07-03 (serve/prepare/bundle over REMOTE_PROTOCOL.md). Open source: self-hostability is a git-grade adoption property. Equal in standing to the serverless head — teams self-host with it for real use; the hosted service builds on the AWS head.
  • forklift-aws-lambda (serverless head) — Lambda handler(s) for the hosted service (possibly several, one per API surface). Named for its cloud on purpose: a serverless head is an adapter to one provider's runtime, so a future Azure/GCP head is a sibling crate, not a feature flag. Currently a stub.

Structure: one cargo workspace in this repo — not separately published crates. Lesson already learned from rust-cli-core: a private-registry dependency costs a publish cycle per change and complicates CI; workspace members share atomic commits, one lockfile, and one CI run. The physical split landed 2026-07-03 (pulled ahead of the Phase 3 remote work so the CLI redesign and the head crates start from the final shape): crates/forklift-core + crates/forklift carry all current behavior; the server and lambda heads are stubs reserving the shape. Dropping rust-cli-core also removed the CodeArtifact registry config and the AWS credentials from CI.

Licensing (settled 2026-07-03): every head is open source (MIT OR Apache-2.0) — core, CLI, server head, and the AWS serverless head. The earlier "serverless head may be proprietary" option is closed: the head is the adapter, and adapter code was never the moat. Anyone can take forklift-aws-lambda and host forklift on their own AWS infrastructure, exactly as anyone can deploy forklift-server on a box. The defensible moat is the operated service — the infrastructure, registry, UI, billing and scale built around the head (Máté's hosting product does this first, in its own repo).

3.5 Configuration & operator identity (implemented 2026-07-02) Done

TOML, two scopes, warehouse wins per key:

  • Warehouse: .forklift/config/warehouse.toml, created (commented template) by prepare. The config folder reserves pallets/ for future per-pallet configuration files — keeping pallet configs in a subfolder means pallet names can never collide with the warehouse file.
  • Global (per user): ~/.forkliftconfig — deliberately not ~/.forklift/…, because a .forklift folder marks a warehouse root and would turn the home directory into a warehouse. Overridable via FORKLIFT_GLOBAL_CONFIG (used by tests; doubles as a relocation escape hatch).
  • Command: forklift config [--global] [<key> [<value>]] — list / get / set. Reads resolve warehouse → global; unknown keys are rejected (a typo'd set must not silently do nothing). Known keys today: operator.name, operator.identifier.
  • Implementation: toml_edit (format-preserving — hand-written comments survive config set). config_utils::get_operator() is the API stack will call for parcel authorship; it errors with setup instructions when the identity is incomplete.
  • Format policy: TOML forever (§6 verdict on Nickel/Dhall/Pkl stands).

4.1 Why this fits Forklift's data model

  • Immutable content-addressed objects are S3's ideal workload. Write-once, never modified, addressed by hash — no coordination needed for the data plane at all.
  • Clients move bytes directly to/from S3. Lambda's 6 MB payload and API Gateway's 10 MB limit aren't obstacles — they force the correct design: the control plane exchanges hash lists and presigned URLs; S3 does the heavy lifting. This is the thing git fundamentally can't do (pack negotiation needs a stateful server), and it's Forklift's core serverless advantage.
  • Parallelism-first pays off remotely too: S3 sustains thousands of requests/second; a client that fans out object GETs/PUTs across cores and connections saturates bandwidth without any server-side process.
  • Per-directory trees/shards mean partial fetch is natural: a Lambda (or client) can resolve a path via the tree metadata without walking the whole tree.

4.2 The lift (push) protocol Spec + server head 2026-07-03

Implemented 2026-07-03: the protocol is specified in docs/format/REMOTE_PROTOCOL.md (version 2026-07-03) and implemented twice-over in this repo: the client engine in forklift-core::util::remote_utils (behind lift / lower / franchise — franchise is the clone verb: you open a local franchise of a remote warehouse), and the self-hostable server head forklift-server. The two server heads are equally valid ways to host a warehouse for real use (decided 2026-07-03): teams self-host with forklift-server; the hosted service builds on the AWS serverless head. forklift-server serves bytes itself where the AWS head will hand out presigned S3 URLs (the spec allows 307 redirects on object endpoints, so the same client speaks to both). Every invariant below holds in forklift-server today: hash verification before an upload becomes fetchable, CAS-only ref updates (fast-forward only, no force push), and — on trusted warehouses — the full audit (office chain + parcel signatures, shared code with the audit command via audit_utils) before any ref moves. The sketch below remains the plan for the AWS control plane.

1  client → API:  POST /warehouses/{id}/lift
                  { pallet, new head parcel hash, ancestor tips, object manifest [hash, size, type] }
2  API/Lambda:    auth user → check manifest against S3 → return missing hashes
                  + presigned PUT URLs into the STAGING prefix + an upload session id
3  client → S3:   uploads missing objects in parallel (staging/{session}/{hash})
4  verify:        S3 event → verifier Lambda streams each object, checks Blake3(content) == claimed hash,
                  then COPIES to the canonical objects/{hash} key. Objects over a size threshold
                  are queued to ECS instead. Nothing unverified is ever fetchable at a hash key.
5  client → API:  POST /lift/{session}/commit
6  API/Lambda:    verify parcel signature (FORK-13), privileges (FORK-10), tree completeness
                  → DynamoDB conditional write: pallet head = new hash IF head == expected old hash
                  → 409 on race (client rebases and retries) — this CAS is the ONLY mutable state

Lower (pull/clone) is the mirror: the API returns the pallet head + a walk manifest (or a bundle reference, §4.3); the client fetches objects in parallel from S3/CloudFront via presigned or public URLs and verifies every hash locally.

4.3 Bundles — the packfile answer Spec + implementation 2026-07-03

Implemented 2026-07-03: the versioned format lives in docs/format/BUNDLE_FORMAT.md (header line + one zstd stream of kind/hash/length/payload records; object records are hash-verified on import, signature sidecars travel as their own record kind). forklift-server bundle builds the whole-warehouse bundle served at GET /v1/bundles/latest; franchise imports it first and fetches only what it lacked as loose objects. The ECS periodic builder and the trailing index for ranged fetches remain hosted-service work.

Loose-object-per-GET works and is cheap (100k GETs ≈ $0.04), but cloning a large warehouse as 100k+ individual requests wastes wall-clock on request overhead. Plan: an ECS job periodically (or on-demand) builds bundles — a single zstd stream of many objects with an index — stored in S3. Clone = fetch bundle(s) + fetch loose objects newer than the bundle. This is git's packfile insight adapted to object storage, and it's precisely the "long-running things on ECS" workload. Bundle format needs its own versioned spec before Phase 3.

4.4 Format properties the serverless target imposes (locking these in)

  1. Objects must be individually verifiable — hash covers the full uncompressed object. Holds today
  2. No object may require another object to be parsed (headers self-contained), so verification Lambdas work object-at-a-time. Holds today
  3. Refs must be tiny and CAS-able — a pallet head is just a parcel hash + name; lives in DynamoDB, not in object storage. Local format done (§3.3) Remote: the CAS ref update of REMOTE_PROTOCOL.md (2026-07-03)
  4. History must be walkable top-down (parcel → parents, parcel → tree → subtrees) so lower can stream breadth-first without an index. Holds today
  5. Everything except pallet heads is immutable. Post-metadata (FORK-9) respects this: it references parcels, never mutates them. Design invariant

4.5 The road to production servers (recorded 2026-07-03) Planned

The 2026-07-03 remote stack is protocol-complete and enforces every invariant of §4.2 (hash verification before anything is fetchable, CAS fast-forward-only refs, full audit on trusted warehouses). What separates it from a full production deployment is performance, scale and operations — nothing below changes the architecture or the wire protocol's meaning. This section is the recorded checklist for both server heads.

Why not a binary control protocol first: considered and rejected 2026-07-03. JSON only carries the control plane (handshake, negotiation, ref updates); object bytes and bundles are already binary. Git's compactness comes from packfiles and delta compression — data-plane properties (its negotiation is hex-hash text lines too) — and the hosted target is JSON-native (API Gateway/Lambda; binary bodies get base64-taxed, which is exactly why bytes go via presigned URLs). The wins below are where the actual bytes are.

4.6 The AWS serverless head — owned by this repo (decided 2026-07-03) Testable spine shipped 2026-07-06 AWS SDK layer next

The decision: forklift-aws-lambda is implemented here, open source, as the serverless counterpart of forklift-server — the AWS "driver": Lambda functions speaking REMOTE_PROTOCOL.md against S3 + DynamoDB. A hosting product deploys it and builds infrastructure, registry, UI and billing around it, exactly as a team deploys forklift-server and builds around that. The head ships the functions and their logic; the IaC (API Gateway wiring, buckets, tables, event rules, ECS) belongs to whoever hosts — this repo documents the required resources, it does not prescribe the deployment tool.

Sequencing (decided 2026-07-05): the AWS head is not built incrementally alongside each feature. Because both heads are thin adapters over one forklift-core and one REMOTE_PROTOCOL.md, feature work stays core-first on the self-host head (fast, fully testable), with an explicit serverless design-gate on every protocol change (statelessness, CAS-on-refs, presigned byte plane, /tmp-only scratch). The AWS head is then brought up as one focused milestone once the features that define "server at scale" — FORK-9 post-metadata and §7.7 optimistic-lift / fleet-remote — settle the protocol surface, and before the hosted product needs it; from that point new protocol features land on both heads together. Rationale: avoids both the double test-tax of co-developing the hardest-to-test head against a moving protocol, and the big-bang late-port risk (the design-gate is the real mitigation for the latter). Revisit if the hosted product's timeline moves up.

The sync/async seam (R4, decided 2026-07-09): the two store traits are synchronous, and the async boundary lives in the runtime adapter, which runs the whole Head call inside spawn_blocking — the same move forklift-server already makes for its handlers' storage work. The audit a store feeds is forklift_core, synchronous to its roots and scoped by a thread-local storage root that must never cross an .await; an async trait would force either a second, async implementation of the security-critical verification path, or a task that resumes on another worker and resolves storage paths against the wrong warehouse. An SDK-backed store therefore drives each future through blocking::AsyncBridge, which requires a multi-thread runtime (on a current_thread one a bridged call cannot drive the IO driver, and would hang rather than fail).

Component sketch:

  • Control-plane function (lambda_http): serves the /v1 endpoints. Object GET/PUT answer 307 presigned S3 URLs (the redirect room is already in the protocol spec); everything else is the same JSON control plane the server head speaks. Multi-warehouse by construction — the warehouse id is a path prefix (/warehouses/{id}/v1/…, the same shape as the server head's planned multi-warehouse mode in §4.5), carried inside remote.url so clients need nothing new.
  • Refs in DynamoDB — the single consistency point: partition key = warehouse id, sort key = pallet#<name> (plus a trust item per warehouse); the CAS is a conditional write on the head attribute. Tiny items, exactly what §4.4 rule 3 demands.
  • S3 layout per warehouse: objects/<hash> (canonical, immutable), staging/<session>/<hash> (unverified uploads), signatures/<hash>, bundles/….
  • Verifier function: S3 staging event → stream the object, check Blake3(content) == claimed hash, copy to canonical, delete staging. Objects over a size threshold queue to ECS (the hosted infra provides the queue/cluster; the head provides the verification code both run).
  • Verification reuse: ref updates run the same audit_utils checks as everywhere else. Parcels, signatures and office metadata are small — the function mirrors them into a warehouse-shaped cache in Lambda /tmp (warm containers amortize it) and reuses the existing filesystem-based verification; tree-completeness existence checks go straight to S3 (HEAD), which needs a small core addition: closure verification with a pluggable existence check.
  • Testing without AWS: the head's storage/ref access goes through two narrow traits (object storage, ref store) with in-memory fakes, so the whole protocol test suite runs in CI against the real handler logic; the AWS SDK implementations take an endpoint override so LocalStack/MinIO can exercise the SDK paths. First deployment against real AWS is the hosting product's shakeout.

Open design points to settle before/while implementing:

  • Staged-verify vs. the synchronous PUT semantics of protocol v1. The server head verifies uploads inline, so a PUT that returns has made the object present; with S3 staging the verification is asynchronous. Either the control plane verifies small objects synchronously at a completion callback, or lift grows the §4.2 session-commit step (POST /lift/{session}/commit waiting on verification) as an additive protocol extension. Decided 2026-07-06 — both, split by size: the additive POST /lift/{session}/commit is the general path, and it verifies the small control-plane objects (parcels, trees, signatures) synchronously at commit — their hashes are re-checked from the store before the ref update, so a corrupt one staged straight to S3 stops the lift — while large blobs, hash-verified asynchronously by the staging verifier, are only checked for presence at commit. The endpoint is additive (REMOTE_PROTOCOL.md); the server head keeps its synchronous inline PUT and never needs it. Shipped as Head::commit_lift in the spine.
  • Auth hook shape. The head validates requests but must not hard-code a vendor's auth: a pluggable authorizer boundary (API Gateway authorizer or a token-validation trait) so the hosting product plugs in real auth and self-hosters can use a static token like the server head.
  • Publishing: whether the functions ship as one binary with a route switch or one binary per surface (control plane / verifier) — cold-start size vs. operational simplicity.

Shipped 2026-07-06 — the testable spine (milestone 1 of the head). The forklift-aws-lambda crate now holds the head's protocol logic, generic over two narrow traits — ObjectStore (the S3-backed byte plane: exists=HEAD, get/put_verified, signature sidecars, and an access/put_target that answers either the bytes directly or a 307 to a presigned URL) and RefStore (the DynamoDB-backed consistency point: pallet heads, an atomic compare_and_set_head, an explicit list_refs since object storage has no directory walk, and the one-way trust door). Head<O, R> implements every /v1 endpoint over the traits — handshake, negotiation, object/signature get·put, the trust door, and the CAS ref update with its full closure/fast-forward/audit gate — plus the additive commit_lift. The whole protocol suite runs in CI with no AWS against in-memory fakes of the two traits (and a redirecting fake that exercises the presigned-URL branch): a real warehouse is built with the CLI, harvested, and replayed against the Head, covering the untrusted CAS guards, a trusted lift that reuses the office + pallet audit, a withheld-blob 422, signature immutability, the redirect branch and the session-commit corruption catch. Verification is reused, not reimplemented: the ref-update handler mirrors the small objects the audit reads (parcels, trees, office records, signatures) into a throwaway on-disk .forklift (scratch.rs) and runs the exact audit_utils checks the CLI and server head run; the one seam a serverless head varies — the working-blob existence check the closure walk makes — is routed to ObjectStore (an S3 HEAD) through a new audit_utils::verify_parcel_closure_with (the "pluggable existence check", the only core change besides a public signature-validator). Next (milestone 2): the AWS SDK implementations of the two traits (S3 + DynamoDB, endpoint-override for LocalStack/MinIO), the lambda_http control-plane entrypoint + the verifier function, and the presigned-URL issuing; the design-gate held — no protocol change was needed, only the additive session-commit.

4.7 Peer-to-peer topology — the server as a first-class peer (recorded 2026-07-08) Exploratory — needs design

The question. Git is already distributed — every clone is a complete repository, and a "central remote" (GitHub) is a social convention, not a technical requirement: bundles, ssh://, git daemon and file:// remotes already let peers sync with no server, and Radicle is the serious prior art (gossip-replicated git with signed peer identities and peer-to-peer issues/PRs). So — could Forklift offer a genuine peer-to-peer mode, peers syncing directly with no central warehouse? Recorded from Máté's question, with his framing of the answer.

Why Forklift is better positioned than git. The parts that make P2P hard in git are the parts git bolts on and Forklift builds in. Identity is self-certifying — a key id is Blake3(pubkey) (sign_utils), every parcel is Ed25519-signed, and authorship is the signature, never a stored field: you trust signatures, not the transport (exactly Radicle's model, already native). Merge is origin-indifferent — merge_utils (three-way diff3) operates on parcel hashes and does not care where objects came from; only the fetch of the bytes is server-shaped. Bundles are already a working offline P2P transport — a bundle_utils stream is self-contained and every record is hash-verified on import, so peer→peer over any channel (USB, LAN, e-mail) works today, just not wired as a mode. The meta-DAGs converge conflict-free (@manifest/@haul merge as a union of signed events), the property a gossip network wants, and audit is offline (audit_utils/verify_pallet_history need no server). The server core already sits behind trait seams — ObjectStore + RefStore + Head<O,R> — and a peer is just a warehouse that exposes those.

What a real P2P mode would have to add (the honest gaps). (1) A transport seam on the client. RemoteClient is a concrete reqwest HTTP struct bound to one base URL — the one real refactor is to extract a transport trait so a peer transport (QUIC/libp2p/TCP, or bundle-over-anything, or gossip) plugs in and a node can be both client and server (the server half is already abstracted; the client half is not). (2) Multiple peers per warehouse — today one remote.url; needs a peer set. (3) Per-peer ref namespacing — the pallet namespace is flat (everyone's branch is just main), so two peers advertising main collide; adopt Radicle's fix (peers/<keyid>/main, each publishing their signed head — Forklift's signed manifests make "whose main is this" cryptographically answerable). (4) Discovery & connectivity — mDNS on a LAN / DHT / relay for NAT; genuinely net-new (git has nothing here either). (5) A merge-authority policy for "no canonical main" — a product decision as much as a technical one; the deferred @haul approval-gating fits perfectly ("merge when N trusted keys sign" is enforceable from signatures alone, zero server).

Topology — the server as a first-class peer (a "seed node"). The most useful shape is not "P2P instead of a server" but a server that is a first-class always-on peer (a "seed node," in Radicle terms) rather than the sole hub — a durable, always-online peer a peer set can lean on for backup, audit and availability. The corporate setting is concrete: a dedicated corporate server peer for backup/audit/compliance, with developers and their agents gossiping locally. And the differentiated angle vs Radicle (human-developer-focused) is agent-native P2P: a fleet of agents with distinct signed identity classes replicating verified work among themselves with no central bottleneck, the corporate server as the durable audit peer — on-thesis for the parallelism-first, agent-native goals.

Two flavors, and the recommendation. (a) Occasional-sync / seed-peer P2P — multiple peers per warehouse + import-bundle-with-merge + a sync <peer> verb: a small delta over what exists (mostly the transport-trait extraction plus a peer list), and the "no canonical main" problem largely evaporates because a designated peer — often the corporate server — is the authority. (b) Live gossip network (Radicle-class) — gossip layer, DHT discovery, per-peer namespaces, always-converging: big, net-new networking, only worth it if (a) proves demand. Verdict: worth doing eventually — specifically flavor (a) — but as optionality, not commitment: do not build speculatively, and do not prioritize it over the §5.0 queue (hardening → AWS head). The cheap, opportunistic move is to keep the seam in mind — when RemoteClient is next touched, an internal transport trait costs little and preserves the option; the full mode waits for a pulling user (offline / air-gapped teams, a LAN agent fleet, regulated no-cloud environments). The hard parts are not the data model (it is ready) — they are discovery/NAT and the merge-authority policy.

Business / monetization implications (hosting-provider strategy, the licensing boundary a peer mode would sit across, cannibalization and sequencing) are out of scope for this design doc and recorded in the private planning notes.

6. Idea review — honest feedback on the backlog

Verdicts: Keep the idea is right as stated · Change right direction, needs a correction · Drop would hurt more than help.

Keep S3 + Lambda + API Gateway + ECS split

The division of labor is exactly right: S3 for immutable data, Lambda for short verifications and negotiation, ECS for long-running work (bundles, big verifications, GC). One sharpening: add DynamoDB for pallet heads. S3 alone can't give you clean compare-and-swap semantics for refs (its conditional writes exist now but listing/metadata/TTL patterns are awkward); DynamoDB conditional writes make the ref update — the only mutable state in the whole system — a one-line CAS. Everything else in the architecture stays coordination-free because refs are the single consistency point.

Change Uploads must be verified before they're trusted — design this in from day one

Presigned PUT URLs mean clients write to S3 unverified — a malicious or buggy client can upload bytes that don't match the claimed hash. If that object lands directly at objects/{hash}, every future client that fetches it gets poisoned content and the dedupe check means it's never rewritten. The fix (baked into §4.2): uploads land in a staging prefix, a verifier (Lambda; ECS for huge objects) streams and re-hashes, and only verified content is copied to the canonical key. This costs one extra copy per new object and is non-negotiable for a content-addressed store. Blake3 being extremely fast (and parallel) is what makes serverless verification economical — your hashing choice pays off here.

Change FORK-10 (per-file privileges / hidden files) — as written, it leaks secrets

The idea of files that some users can't even see is genuinely differentiating (git has nothing like it). But two hard problems must be solved in a design round before implementation:

(1) Verifiability: if a restricted file is fully omitted from a user's copy, they can't reconstruct tree hashes, so they can't verify history or build parcels on top. Fix: keep the tree entry (name + hash) visible, withhold only the blob content.

(2) The hash itself leaks. Your example is a .env file — exactly the worst case: secrets have low entropy, and a visible Blake3 hash lets an attacker confirm guesses offline (hash "AWS_KEY=AKIA…" candidates until one matches). Publishing the hash of a secret is a dictionary-attack oracle. Fix options: restricted files get keyed hashes (Blake3 has native keyed hashing — hash with a per-warehouse or per-file secret key), which breaks dedup for those files (fine — they're secrets), or store them encrypted with the hash computed over ciphertext. Either way this feature needs its own spec. Verdict on the idea: keep it, but it is the hardest feature in the backlog — schedule it last (Phase 4) and don't let earlier format decisions foreclose the keyed-hash option (the blob format's version byte gives you the room — a "keyed blob" can be a new blob version).

Change FORK-13 (signed parcels) — great design, two tightenings

The genesis-parcel trust anchor is sound — it's the same shape as certificate-transparency roots, and making user management itself signed parcels means the entire trust history is auditable offline. Two changes: (1) the ticket says to hash "at least important data like parent commit ID, timestamp, user info" — don't select fields; sign the parcel's object hash. It already transitively covers the tree (all content), parents (all history), and every metadata byte; field-selection creates malleability bugs. (2) Your own note in the ticket is right — reject the crossed-out "temporarily disable signing" escape hatch entirely. Signature-required should be a one-way door, with admin-lockout handled by archive + re-genesis as you described. Also plan a small addition early: parcels should record the key id used (you have this) and the service should support key revocation records as tracked metadata — compromise recovery is the part first drafts always miss. Algorithm: Ed25519, no reason to look further.

Keep FORK-9 (post-metadata parcels) — with one structural guardrail

Storing approvals/merge records as tracked files snapshotted by special parcels is one of Forklift's best ideas: the "GitHub layer" becomes part of the repository — portable, auditable, and it survives moving hosts. Guardrail: keep post-metadata parcels on a dedicated metadata pallet (or an explicitly separated parcel type excluded from normal log/merge views) rather than interleaving them with code history — otherwise every approval doubles the visible history and confuses merges. The immutability invariant from §4.4 already covers the rest: post-metadata references parcels, never modifies them.

Change FORK-4 (guess command) — suggest, don't auto-execute

"If there is exactly one closest result, go with that" is a footgun in a VCS: a typo'd command silently running a different command is exactly how people lose staged work (imagine a typo resolving to unload). Do what git does: print Unknown command "laod". Did you mean "load"? and exit non-zero. Auto-execution could be opt-in later behind a config flag for the brave, but the default must be suggest-only.

Drop Nickel / Dhall / Pkl config support (FORK-1 future note)

TOML forever is the right call; programmable config languages multiply parsing dependencies, error-message complexity, and docs burden for near-zero benefit in a VCS config that holds a repo name and user identity. Git's dead-simple config is a feature. If richer config is ever truly needed, that's a signal too much policy lives in config.

Keep FORK-25 (git aliases) and FORK-26 (professional mode) — small sharpening

Git aliases are a cheap, high-value adoption lever — a git user should be productive in minutes. For professional mode: instead of (or in addition to) Ok/Err: … strings, add --json structured output per command. Scripts and future tooling (your hosting service's runners, editors) want machine-parseable output, and "Ok" strings inevitably grow ad-hoc fields. The exit code already carries success/failure since the fix pass.

Keep Dock = inventory (FORK-16, resolved 2026-07-02)

Adopted reading: the dock is the sharded inventory — one staging representation, extended with repopulate-on-shift and compare-on-stack. Building a second index alongside the inventory would duplicate state and reintroduce the consistency problems just fixed. Open cosmetic question: rename "inventory" to "dock" for terminology charm (forklifts load pallets at a dock — it's honestly a better warehouse word). Costless now, annoying after the format docs multiply.

7. Forklift × AI coding agents (recorded 2026-07-03) Strategy — pick & schedule

The bet: AI coding tools (Claude Code, Codex and their successors) are becoming the primary consumers of version control, and they change what a VCS should optimize for in five ways: (1) authorship becomes a compliance question — what wrote this, under whose authority; (2) the CLI's consumer is a program with a token budget, not eyes; (3) concurrency goes from team-size to fleet-size; (4) history splits into a noisy trail (checkpoints) and a clean story (reviewed changes) that both must survive; (5) workspaces become ephemeral, parallel and task-scoped. Git treats all five as retrofits (freetext authors, porcelain scraping, worktrees, squash-and-lose-the-trail, sparse-checkout pain). Forklift can make them primitives — and the moat features build directly on what is already shipped: the signed office (§3.3/Phase 2), tracked metadata (FORK-15), the sharded inventory, and the stateless remote (§4).

An agent is admitted to the office like any operator, but marked as one: office admit --agent --supervisor <operator>. Design cost: small — agent/supervisor fields on the office records + filters in history/audit.

Key isolation is the load-bearing part (decided 2026-07-05). The office marking makes authorship legible ("an agent, supervised by Alice"); it does not by itself stop an agent from signing as Alice, because forklift keys are unencrypted files and signing is non-interactive — any process running as the user can use them. So the marking must be paired with a mechanism that keeps the human's key out of the agent's reach. Ideal is OS isolation (separate user / sandbox / hardware key). The shippable software mechanism, Stage 1 done 2026-07-05: passphrase-protected human keys — office enroll/keygen/rotate --passphrase encrypts the private key at rest (Argon2id → ChaCha20-Poly1305); signing prompts on the terminal, so an unattended agent with no passphrase fails closed and cannot sign as the human. Agent/bot/CI identities stay passphraseless by design (autonomy is their nature) — the split is: a human identity requires a per-action human gate, an automated identity signs freely under its own marked, scoped, revocable key. Principle: automation signs as itself, never as a human; the passphrase is what forces that. Honest limit: a passphrase defeats an honest/semi-trusted agent, not a fully malicious one with shell (it could trojan the binary or capture the passphrase) — that needs OS isolation or hardware keys. Core never prompts: it exposes a passphrase-provider seam the CLI head fills (sign_utils::set_passphrase_provider).

Agents checkpoint constantly (content-addressed dedup + parallel Blake3 make that nearly free); humans want one clean reviewed parcel. deliver consolidates a draft pallet's checkpoint chain into a single signed parcel on the target pallet and retains the full trail as post-metadata referencing the (kept) checkpoint parcels — excluded from normal history views. Shipped 2026-07-05 7.5 Bays — parallel working directories as a primitive

A warehouse has many loading bays: bay <name> creates an additional working directory bound to the same warehouse — shared object store and refs, per-bay inventory, current pallet and lock. Fleet-on-one-box is the shape agent tooling actually runs in.

Done 2026-07-11 — design settled 2026-07-10; all six stages shipped 2026-07-10/11 7.6 Task-scoped sparse workspaces

Give an agent only the subtree its task needs: franchise --only src/service-x (or a scoped bay) materializes one directory subtree — per-directory trees make partial fetch natural (§4.1 noted this from day one), and the sharded inventory means the rest of the warehouse simply has no shards. Smaller blast radius, less context to load, much faster cold-start for ephemeral cloud sandboxes. Git's sparse-checkout is notoriously painful and enforces nothing. The design round (2026-07-10, adversarially reviewed — see the changelog entry for what the review broke) settled the following, staged as: scoped bays on a full store first (materialization-only, no fetch, no protocol — the cheapest, most testable slice), then sparse franchise (fetch + materialization sparse, the ephemeral-sandbox headline).

Hash-sealed but unfetched, user-pallet content only. The object store holds, for every parcel it knows, the spine of tree objects down to each in-scope path plus the in-scope subtrees/blobs; it omits out-of-scope subtree objects and blobs. Every omitted subtree's hash still lives inside a spine tree object the workspace holds, committed by the signed parcel — out-of-scope content is cryptographically pinned to an exact hash it hasn't downloaded. Meta pallets (@office, future .forklift/meta/*) are never scoped: the office chain reads full tree+blob content at every audit step (read_office_state_of, called from verify_office_chain), so scoping office fetch would silently break office audit. This is a hard invariant, not a choice — franchise/lower route office and other meta pallets through the existing, unmodified, unscoped fetch/audit path unconditionally; only the user pallet's content tree is ever path-pruned.

The head-derived overlay stack. stack rebuilds the root tree as "head tree (or the merge-resolved skeleton) with the in-scope subtree(s) freshly built and spliced in," byte-identical to a full build — no stub shards, no stub objects: the out-of-scope hashes already live verbatim in the head's spine, so the scope file plus a head-derived splice is the one source of truth (stubs were rejected as a second copy of that hash that can drift after a lower/consolidate moves a sibling). The overlay must replicate build_tree_from_inventory's empty-subtree pruning recursively up the spine (an ancestor that becomes empty because its only child was pruned is itself pruned from its own parent) or a sparse stack's tree hash diverges from what a full workspace would produce for the same content — this is what makes "a scoped bay and the full main tree stack the same tree hash" actually hold.

Scope is local, two-layered, three-valued, and never touches meta pallets. Warehouse fetch scope (shared, bounds every bay) vs. bay materialization scope (bay-local, always ⊆ fetch scope), both local-only path-prefix records, never tracked. The classifier is not a boolean: a path is Spine (a strict ancestor of an in-scope path — walked, but its other entries at that level are out-of-scope and copied by hash, never descended), InScope (fully materialized, walked normally), or OutOfScope (sealed, never loaded). A directory↔file type change observed at a Spine path (nothing stops a later revision from replacing a directory entry with a same-named file entry or vice versa) refuses the operation (scope_path_type_changed) rather than guessing — the Spine's entire job is "carry the hash forward assuming it's still a directory," and a type flip breaks that assumption in a way a sparse workspace cannot safely reason about without loading the object.

Merge: hash-only three-way resolution, and it reaches file/symlink entries too. For each out-of-scope entry — subtree, file, or symlink alike — a one-sided change (O==B or T==B) resolves by hash, no blob loaded; a genuine two-sided out-of-scope conflict refuses (out_of_scope_conflict, next-step: widen scope or resolve in a full workspace). This closes a real gap: the existing subtree-level merge walk has no O==B fast path (only file-level merge did), and the existing file-level fast path routes to a normal blob apply, which is wrong once the file is out-of-scope and the blob is absent — both need a classify-before-dispatch gate ahead of every blob load. Resolved-by-hash out-of-scope entries land in a small side skeleton (kept beside .forklift/consolidation), consumed by the next stack as the overlay's base instead of "current head."

The mandatory parent-set-aware lift closure walk. A sparse merge can legitimately adopt an out-of-scope sibling by hash from its second parent — that merge's tree then carries an out-of-scope hash that differs from the first parent's value at that path. lift's negotiation-pruning closure walk compares a new parcel's tree only against its first parent's tree; against a merge like this it wrongly decides the subtree changed and tries to load the absent object. A sparse workspace could consolidate such a merge but not lift it — this was the one broken correctness claim the adversarial review found. Fixed by generalizing the walk's base from "the first parent's tree" to "every parent's tree," skipping when an entry is explained by any of them (an ordinary single-parent parcel is just the N=1 case, no separate code path). This keeps scope entirely out of the remote/lift layer — the walk still only ever proves "the remote already has this because some ancestor it already has explains it," it just checks all ancestors instead of one — and is a real correctness/efficiency improvement for merge lifts in full stores too, not a sparse-only patch.

Zero-protocol MVP; the FORK-10 seam is additive. Sparse fetch is a path-pruned version of the existing fetch_history walk over existing endpoints (objects/{hash}, objects/batch) — no wire change. A sparse franchise skips the whole-warehouse bundles/latest and uses the batched in-scope walk instead. Enforcement is advisory today — a client choice, not a security boundary; git sparse-checkout's "enforces nothing" gap is real here too until file-level FORK-10 lands. The design specifies, without building, the eventual seam: a path-addressed fetch endpoint (hash-addressed fetch is path-blind — the remote can't authorize by path without one), additive, wired to the FORK-10 authorizer when it ships. Confidentiality (hiding content, not just not-fetching it) is a separate feature (the locker private tier, §7.9, keyed hashes) — §7.6 must not be sold as confidentiality. A sparse workspace's out-of-scope closure is only proven present against the remote it was actually scoped/fetched against; lift to a different remote refuses client-side with a clear message rather than fail late and confusingly server-side — full multi-remote federation is deferred.

Narrow vs. scope-prune stay separate. Dropping a path from a bay's materialization scope is bay-local, safe, and frees nothing shared — the underlying objects remain reachable from pallet heads (ordinary history, not garbage), so reachability-gc correctly does not, and structurally cannot, free them. Actually reclaiming shared disk is a distinct, later, explicitly destructive, multi-bay-aware scope-prune verb (must intersect every bay's current scope, not just the invoking one, and atomically narrow the shared fetch-scope declaration so post-prune absence reads as "sealed" not "missing") — deferred, never folded into gc or into narrow.

The sparse-audit trust statement, stated precisely. Signatures are fully verified (parcel history + the office chain, identical strength to a full clone — office is always fetched in full per the carve-out above). Trees are re-hashed on every read. Blobs are only presence-checked, never re-read for content — true of full-store audit too, not a sparse regression, but stated exactly rather than implied stronger: a present-but-content-corrupted in-scope blob passes the closure check and is caught on next actual read, not at audit. Out-of-scope subtrees/files/symlinks are sealed by hash — not verified for availability, only pinned so they can't be forged or substituted, and verified for real content only if/when scope later expands. A sparse audit's success output, human and --json alike, must never be able to be mistaken for a full-clone "verified" line — a distinct message and a --json scope field are mandatory whenever the audit ran sparse.

Recorded 2026-07-08 — needs design 7.9 Locker — per-user tracked files with their own visibility (a friend's idea)

The gap. Developers and their agents accumulate files that belong with a repo but not in it — most sharply an agent's .claude/ (skills, subagents, MCP config, settings), plus editor prefs and personal notes. Git has no home for them: .gitignore + a disconnected dotfiles repo, or skip-worktree/assume-unchanged hacks (local, unshared, unversioned). Nothing lets you version personal files alongside a repo without forcing them on collaborators or dragging in theirs.

The feature. A per-operator locker (the warehouse metaphor made literal — a worker's locker, where you keep your own things while on the job, never part of the shared inventory): a tracked, signed namespace owned by one operator (addressed e.g. @me / @user:<id>), carried by the repo but governed by the owner's visibility & pull policy, in three tiers: (1) private but tracked — only the owner fetches/materializes it, withheld from everyone else; (2) readable, not auto-pulled — others can see and read it, but a lower never drags it into their working tree; (3) opt-in pull, whole or partial — a teammate can choose to pull a locker, or just parts of it.

Why it fits — a composition of primitives Forklift already has or has designed, not a new subsystem: meta-pallet namespaces (§3.3) give the locker its own ref namespace so it never interleaves with or complicates main history and merges; per-file privileges / FORK-10 (§6.3) give the private/readable tiers, reusing FORK-10's verifiability answer — keep the tree entry (name + hash) so others still verify history, withhold only the blob; task-scoped sparse fetch (§7.6) gives "don't auto-pull" and "pull just parts" (per-directory trees make partial selection natural); signed authorship (Phase 2) means only the owner writes their locker and a teammate's merge never touches it. The three tiers map one-to-one onto those.

Why it matters (agent thesis). Agent configuration is the emerging dotfiles problem — an agent's skills/subagents/MCP servers/settings are personal, evolve constantly, and you want them versioned with the project they configure yet neither imposed on the team nor entangled with theirs. Git has no answer; Forklift's namespace + visibility model does. A concrete, resonant demo of the whole agent-native pitch — a candidate launch highlight, not just backlog.

MVP vs full. The readable, sparse-by-default tier is buildable on meta-pallets + sparse fetch alone (no cryptography) and captures most of the value for low-sensitivity config — ship it first. The private tier (content that must not leak) inherits FORK-10's hash-leak problem (§6.3: a visible Blake3 of low-entropy content is a dictionary oracle) and so lands with FORK-10's keyed-hash work. Open design questions: locker addressing (@me vs @user:<id>), how the visibility/pull policy is recorded and server-enforced, how partial-pull selection is expressed, and whether a locker is per-operator across all bays (§7.5).

8. Identity, access & the hosting-provider layer Design recorded 2026-07-04 Core primitives: this repo · Provider: separate product

Thesis. Forklift stays the git layer — signed, offline-verifiable content authority that lives in the warehouse. A hosting provider layers global identity + access on top, GitHub-style, and never puts content authorization in its own database. Identity goes global (provider, soft, mutable); content authorization stays on-chain (per-warehouse / per-org, hard, forge-proof). This section records the full model designed on 2026-07-04. §8.14 splits it into Forklift-core work vs provider work and lists the decisions still open. It extends the Phase 2 trust stack (FORK-13) and the pallet-level office (FORK-10); it does not replace them.

8.1 Two layers, two axes

Git has no users; GitHub layers accounts on top and never stores access control in the repo. Forklift is the same shape: the per-warehouse office is the git layer (already built), the provider is the GitHub layer (new). The mistake to avoid is collapsing them. Concretely there are two axes that must not be conflated, because they have different owners, tenses, and change-costs:

Permission vs attestation. A provider grant is a present-tense permission ("may alice write, now?"); an office entitlement is an append-only attestation ("this signed content was authored within entitlement, then"). A demotion revokes the permission and correctly leaves history intact — you never rewrite the past. They are not two copies of one fact to keep in sync.

Rule — single owner per axis. The provider owns access; the office/chain owns content authorship; the provider writes into the chain, never the reverse. Two sources of truth that can disagree is the bug factory.

8.2 Users vs operators — the id seam

  • Users live in the provider DB (DynamoDB) — soft, mutable, high-volume: profile, email, login, org membership. Operators live in the chain — hard, signed. Enrollment is the moment a user becomes an operator in a warehouse.
  • The global identity registry is not a chain (mutable, high-volume, cross-cutting — everything chains are bad at). It is a DB keyed by the operator id the chain sees — the join key between the mutable identity world and the forge-proof authorization world. One user, one id, many orgs. Refined 2026-07-05 The id is an opaque string to the chain, held in the single operator.identifier config value: the provider mints a UUID and hands it to the client; standalone clients auto-mint a UUID when nothing is configured (pseudonymous by default, no policing needed); a team may deliberately set a readable id, accepting it is public in every clone forever. Display name is separate local/provider data with its own lifecycle — a handle can be fixed, the id must never drift.
  • The key set is the exception: the binding uuid → keys must be a signed sigchain (§8.5), not soft DB rows — otherwise the provider could add a key and forge. The DB caches it; the signatures are the authority. Profile/membership stay soft; the key set is hard.
  • Account-lifecycle audit (create/suspend/rotate) becomes provider-trust (DB + CloudTrail), not forge-proof. Optional upgrade: a transparency-log digest (a signed Merkle root over the account event log) gives tamper-evidence without putting mutable state on a chain.
Principle — users matter when they touch content. On-chain footprint is proportional to content actually touched. Enrollment is lazy, per-warehouse, at first grant; a user who never authors has zero chain presence. Anything that touches a warehouse's content or its authorization is a signed chain event (audited, forge-proof); pure account lifecycle is DB state. The chain audits the exercise of authority, so past touches can't be forged no matter what the DB later says: the DB is soft/rebuildable, the chain is the standalone truth.

8.4 Owners & namespaces — handling "outside orgs"

The top-level unit is an owner (a namespace), of which org and personal account are two flavors — a personal account is an org-of-one. Crucially an owner is a cheap DB record, not a chain, so millions of them scale fine; chains exist per-warehouse (and per-org) and are created lazily. That dissolves "outside orgs":

  • Above orgs (site admin, org lifecycle) → the meta chain (§8.3). Not "outside" — the ceiling.
  • Solo / personal → an org-of-one. Zero special cases; "per-org" is really "per-owner".
  • Cross-org collaboration → the key is the border-crossing unit, not a global identity. A guest is enrolled by having their key added to the target warehouse's office; that enrollment is the recorded audit of the crossing.
  • Public → a per-warehouse visibility flag (reads may skip enrollment; writes never).
  • Bots / CI → non-human keys, enrolled like any principal.

Decision Personal namespaces first-class from day one, or pure B2B v1? Pure B2B lets v1 skip personal namespaces and public visibility ("solo" = a one-member org) and add them later.

8.5 Keys, identity roots & the sigchain (multi-device)

An operator identity is a set of keys (CLI, browser, laptops). The linchpin that makes "the provider can't forge" hold end-to-end: a new key is valid only if an already-trusted key signs "K_new is also me" — the Keybase-sigchain / Signal-device-linking model. Key trust self-propagates from a genesis key; the provider, holding no trusted private key, cannot inject one.

  • The office enrolls the identity root, not individual devices. Add a device → one sigchain entry → accepted by every warehouse automatically, no per-warehouse fan-out.
  • Genesis key generation must be client-side. The provider never sees any private key, ever, including at signup — a provider-generated root is a poisoned root the sigchain can't save.
  • Revocation is a sigchain entry signed by a surviving key. Losing a device is self-service while ≥1 key survives; total loss falls to recovery (§8.7). Residual: the provider can serve a stale sigchain (withhold a revocation) → append-only + transparency log + clients pin the last-seen head.

If the same operator id appears in two orgs' chains, different keys don't stop correlation — the id itself links you. True unlinkability needs per-org operator ids (per-org pseudonyms, with the provider holding the id→account joins in its DB; resolution is server-mediated anyway, §8.12, so display is unaffected). Decision recorded: ship the global-id model now (one id everywhere — the §8.2 join key), and treat per-org pseudonyms as a provider-layer upgrade decided when orgs exist. The client side is already shaped for it: profiles (§8.14 implementation note) let one machine hold several ids with their keys.

8.6 Enrollment & admin-authorized keys

Enrollment is the CSR pattern (certificate signing request):

1  you (client)   generate/hold your identity root key     private never leaves you
2  you → admin    request { pubkey, "join W" }  SIGNED by your key   (proof-of-possession)
                  over a channel where the admin can confirm it is really you
3  admin          verify (a) out-of-band it is you   (b) the self-signature (PoP)
4  admin → office parcel { operator=uuid, identity_root=PUB, role }  SIGNED by admin key
                  (chains to the warehouse anchor)
  Secure channel = right human.  PoP = right key + consent.  Admin sig = authorized here.

Two bindings, only one of them cryptographic. key ⇐ identity_root (same enrolled identity) is guaranteed by the endorsement signatures. identity_root ⇐ human ("this root is alice") is an assertion — anchored out-of-band at signup and at admin enrollment; crypto proves "same key-holder", never "this human", from nothing (GitHub is identical). Proof-of-possession at enrollment (you co-sign your enrollment with your root) closes the gap where a malicious admin/provider enrolls a key you don't control.

Change (corrects an earlier stance) Admins may authorize new keys for existing users — under one principle: the scope of a key-authorization equals the scope of the authorizer's authority. Your own key → global (your whole identity). An admin → scoped to their org/warehouse (an admin cannot write your global identity, or one org's admin could impersonate you in another). The record is { key, belongs_to, authorized_by } signed by both the authorizer and the new key (PoP) — self-service linking has your existing key as authorizer; recovery has the admin. Recommended B2B model: an org-scoped identity record where a new key is authorized by either your existing key or an org admin. Tradeoff: admin-authorization means an admin can impersonate a member within the org — but that is expected enterprise behavior, it is tamper-evident (signed + logged) and notifiable, and strictly better than silent provider control.

Onboarding (recorded 2026-07-05). The CSR line office keygen prints — <operator-id> <pubkey> <pop> — is the onboarding payload in both worlds; the provider changes the transport and UX, never the trust flow. Standalone: enroll the first warehouse (id auto-minted, root self-endorsed); a teammate hands you their keygen line over a channel where you can tell it is really them (that human channel is the identity binding) → admit. Second device: same id in its config, keygen there, link from the old device. With a provider: web signup mints the id (email/login stay soft in the DB) → forklift login writes id + token into a local profile → the client generates the root key client-side and uploads pubkey + PoP → the provider cannot enroll anyone (it holds no signing key, §8.8): it queues the CSR and shows an org admin an approve button, whose click produces the same admin-key-signed admission (via the §8.10 signing tiers). Verified email replaces the out-of-band channel; everything below the button is the shipped primitive.

8.7 Genesis, org non-orphaning & recovery Re-genesis shipped 2026-07-05

  • Self-signed genesis is standard, not sketchy (root CA, git init, genesis block). Its safety is the one-way door anchor: trust starts self-asserted at the genesis instant and can never be replaced — you can't self-sign into an existing warehouse. Root the genesis in the owner, not the individual: personal → self-sign with your key; org → root in the org's authority so a warehouse isn't orphaned to a person's key when they leave.
  • Non-orphaning is an org-governance property, handled cheaply: require ≥2 admins at creation, block removing/demoting the last admin (an office-chain invariant), and warn at one. A per-org signing key is not needed for this; if ever used it is a rarely-touched escrowed root (CA-style), never a shared day-to-day signing key.

Re-genesis (trust reset) is the recovery primitive for a fully-locked chain. It does not create trust from nothing — it relocates "recover lost keys" to "who authorizes a new root", which must come from outside the dead chain:

  · LOUD: old-anchor holders must consciously re-accept (SSH host-key-change / Signal safety-number)

8.8 The provider's honest security boundary

With no standing key enrolled in any office, the provider cannot silently forge within a live chain — only users sign their own actions. What it can do (if it gates re-genesis unilaterally) is a loud, total, visible re-genesis — never a quiet edit. So the honest, shippable claim is prevention of silent forgery + detection of seizure:

Claim you can make: "the provider cannot touch your chain undetectably." Any interference is a conspicuous full reset (anchor changes, old-root holders must re-accept, transparency-logged).
Stronger opt-in (a dial): gate re-genesis behind an escrowed org recovery key / quorum / notice period → the provider genuinely cannot seize — but then it also cannot unilaterally recover a fully-orphaned org. Pick per org.
Teeth: the whole guarantee lives in clients verifying offline against a pinned anchor. "Loud" is only loud because clients pinned the old root; design anchor-pinning in from the start.

8.10 Web-UI signing

The hardest practical problem: a web "merge" button must produce something, but storing usable private keys server-side means the provider can forge. Resolution — two signing modes (GitHub's actual model) tiered by the content-signing dial:

8.11 Offline audit with embedded keys

Keep Warehouses do store keys — self-containment requires it (as today). The change from today: keys in the office are user-signed sigchain endorsements, not provider-asserted rows, and they embed lazily, at authorship time. The office pins the identity root for the role; when a key first authors in a warehouse, its endorsement chain rides along. A warehouse accumulates only the keys actually used in it.

Verify a parcel offline (everything is inside the clone — no provider, no network):
  1  signature by K            valid?
  2  embedded chain  K ⇐ … ⇐ identity_root     (endorsements, all signed)
  3  office grants identity_root the role that authorized this write
  4  all valid AT AUTHORSHIP TIME (per revocations known to this copy)

Revocation is a negative fact, so embed-on-use structurally can't carry it (a revoked key stops being used — nothing to ride in on). It needs push or reference. Chosen approach — the org chain is the CRL, so it travels with the offline copy and also resolves the §8.3 cross-tier references:

  • Member working copy = warehouse + full org chain. Simple, complete, entitled.
  • Guest / external / audit copy = warehouse + a proof-carrying slice (Merkle inclusion proofs against the org head): only the keys that touched this warehouse and their revocations — no org-roster leak.
  • Freshness is relocated, not eliminated: an offline copy knows only revocations up to the org head it holds (fundamental — offline PGP is the same). The online path is unaffected — the server holds the org chain and rejects revoked keys immediately.
  • Refined 2026-07-05 (shipped) Revocations carry a reason + distrust boundary — not an effective-time: validity questions are decided by exact ancestry, never by timestamps (the §3 principle; also immune to a signer shifting their clock). The boundary is the set of pallet heads the revoker vouches for at revocation (the remote's heads included, like enroll's trust boundary); signatures by the revoked key are valid only on parcels reachable from it. Retirement vs compromise is recorded for humans and tooling — the mechanics are uniform, and they also close two holes: a retired key can no longer sign new parcels that verify (previously "retained forever" meant "verifies forever"), and revocation is append-once for everyone, admins included (no quiet un-revoking, no boundary rewrites — a chain invariant like the last-admin rule). A revoked key also cannot extend the office chain or endorse new keys.

8.12 Pseudonymous chains + controlled name resolution

The chain stores only operator UUIDs + public keys — zero PII. This fixes git's worst privacy flaw (names/emails baked into immutable history forever) and yields two strong properties:

  • Verification is name-blind. Integrity/authz run entirely on UUIDs + keys + signatures — names are needed only for human display. Privacy (pseudonymity) and integrity (full offline verification) are orthogonal; you don't trade one for the other. This is also why bundling a pseudonymous org chain (§8.11) is acceptable — it leaks pseudonyms, not names.
  • GDPR-style erasure is possible. Delete the UUID → name mapping (soft, DB) and the person becomes permanently unresolvable while the immutable chain keeps its integrity. Immutability and right-to-be-forgotten at once — something git cannot do.

Honest calibration: pseudonymous ≠ anonymous — a stable UUID is a behavioral fingerprint (timing, which modules, correlation with releases), so a determined observer with chain access can often re-identify without the resolution service (hence slicing, §8.11, still matters). Resolution must be a server-mediated, policy-gated call — never bundled, or the policy is merely advisory. It protects users from each other, not from the provider.

Tiered resolution policy (admin-configurable presets, not baked-in — it is a small resolution policy engine):

TierMay resolve

Decision Is "shared a warehouse" current or historical/contextual? Current-only makes history unresolvable as memberships churn; contextual ("you could resolve them if you shared a warehouse when they authored it") is likely what you want.

  • Admin-authorized recovery keys (2026-07-05). office authorize <operator> <pubkey> <pop> closes the §8.6 recovery gap: an operator who lost every device runs keygen on a new machine and hands the printed line to an admin, whose endorsement authorizes the key — valid exactly in the office they administer (the scope rule the audit was already enforcing; only the emitting command was missing). This completes the §8.14 "proof-of-possession enrollment; admin-authorized key records" row.
  • Hook protocol (2026-07-05). Every request is MAC-signed (Blake3 keyed, derive_key-versioned context, 300 s replay window) — a hook without a secret refuses to start. Every hook is server-invoked — the server head holds the URLs and secrets (a single [hooks] config table; authentication_cache_secs caches positive answers, 60 s default), and a client only ever speaks the remote protocol to the server. Resolution therefore is not a client-side hook (an earlier draft made it one — wrong: a client holding a shared secret cannot enforce the §8.12 tiered policy): the client asks the server via a new additive POST /v1/resolve endpoint, the server authenticates the caller and forwards to its resolution hook (passing the caller so a policy-aware directory can tier), and any failure — no remote, no hook, unreachable — silently shows pseudonyms. history/office list resolve one office-roster-bounded call through the configured remote. Verification/signing paths take no hooks by construction. Wire types in model::hooks + model::remote, MAC helpers in util::hook_utils, the client resolve in util::remote_utils. One enforcement refinement landed with it: office roles bite only once the office has been lifted — between the trust PUT and the first office lift the transport token remains the gate (the office ref update still verifies the chain against the anchor), which is what lets a hook-authenticated operator bootstrap a trusted warehouse.

The key directory gains a key-owner manifest (owners.toml: key_id → operator id) so a machine can tell its identities' keys apart, including keys generated before their admission lands anywhere — local bookkeeping only, the office stays the authority on key↔operator binding. Profiles are also the client-side landing zone for per-org pseudonyms if that upgrade is taken (§8.5 correction).

9.2 The staging area — keep the inventory, reconsider the mandatory step

Context: jj and Sapling both drop Git's staging area — but differently. Sapling (Mercurial lineage) has no index; commit records all tracked changes and partial selection happens interactively at commit time. jj subsumes staging: the working copy is itself a commit, auto-snapshotted, and "staging" becomes ordinary commit editing (split/squash -i). "No staging area" bundles two separable claims: (a) ergonomic — no mandatory explicit stage step; (b) data-model — no persistent middle state. jj keeps (b), drops (a); Sapling drops both.

Decision (2026-07-06). Keep the inventory as infrastructure — it is load-bearing (bounded memory, per-directory parallel walks, the stat-cache fast path, per-bay isolation), and FORK-16 already resolved "the dock is the inventory; don't build a second index." So the question is not "staging area yes/no" but "should load be a mandatory user-facing step?"
  • Honest problem today: Forklift sits in the awkward middle — it imposes the friction of an explicit stage step but offers none of staging's payoff (no hunk-level add -p). Users pay the tax without the feature.
  • Change Consider a Sapling-style auto-stage default: let stack snapshot all tracked changes by default (a commit -a-by-default), keeping load as the optional tool for partial commits. Small change, no data-model impact, removes the friction Git users grumble about while load still exists.
  • Change Add partial selection regardless (hunk/path-level) — a real gap vs. all four alternatives, and the thing that would finally make the staging step earn its keep.
  • Drop Full jj-style working-copy-as-parcel — too deep a change to shift/stack/undo, and it rubs against the signed one-way-door model. Forklift's existing answer to "agents checkpoint constantly" is draft pallets + deliver + auto-checkpoint (§7.3), which buys jj's best robustness property ("can't lose work; undo anything") without rearchitecting the working copy.
Recommendation: do partial selection either way; seriously weigh making the stage step optional rather than mandatory; leave working-copy-as-commit to jj.

9.3 First-class / committable conflicts Needs design

Context: in Git — and Forklift today — a conflict is a transient, out-of-band, blocking state: markers in the working file, a conflict entry in the inventory, and stack refuses until resolved. In jj and Pijul a conflict is a value that can live inside a commit: legal to commit, shareable, resolvable later. This bundles three separable benefits, unequal in depth: (1) non-blocking (operations always complete, conflicts recorded not halting); (2) durable & shareable (commit/push/hand-off a conflict); (3) auto-resolving through rebase (jj's deep magic — conflicts propagate through a rebased stack and self-resolve; tied to jj's automatic-rebase machinery).

Adopt — scoped A conflict is a signed, syncable object

This fits Forklift's object model better than it fits Git's, and it is half-built already. Sketch, staying on-model:

  1. A conflict object via the versioned formats. Formats are VLQ-versioned ("adding a version is a match-arm") and there are already OBJECT_TYPE_CODES/DIR_ENTRY_TYPE_CODES. Add a conflict blob variant storing the sides as content addresses ({base, side_a, side_b, …} — cheap, dedup'd, already objects) and a tree entry type meaning "this path is conflicted → points at a conflict object."
  2. A conflicted parcel is a normal parcel. A conflicting merge builds a tree with conflict entries and stacks a two-parent parcel over it — legal, signed, committable. jj's move, adapted to the snapshot model.
  3. Materialization renders, doesn't own. shift renders diff3 markers into the working file (as today) but the structured object is the source of truth; the existing conflict-state inventory entry is the hook.
  4. Resolution = editing the conflicted parcel. On load/stack, markers gone → the conflict object is replaced by a plain blob (resolved); markers remain → it persists.
  5. Transport is free. Conflict objects are content-addressed → they lift/lower/bundle and hash-verify like anything else — zero server change, exactly the @haul/@manifest pattern (new object kinds, no special-case).

Head start: the conflicts command already reconstructs base/ours/theirs into content-addressed blobs (§7.4). The step from "reconstruct at read time" to "store a durable conflict object" is small. It also subsumes the deferred per-parent-conflict-stages item (§3.3.1) — a conflict object holds N sides natively.

Honest limits: cross-cutting (every tree-walker learns the entry type: merge, materialize, diff, stocktake, stack, audit, export-git); export-git gains a lossy edge (Git has no home for a committable conflict — refuse or flatten to markers); audit must bless conflict-carrying parcels; and benefit (3), auto-resolve-through-rebase, is explicitly out of scope — it needs rebase machinery Forklift doesn't have and shouldn't build for this. Forklift would get the committable half of jj's story, not the whole thing.

Why it's worth it (the Forklift-specific argument): the agent thesis makes non-blocking, durable, shareable conflicts more valuable here than in a human-only VCS — a fleet shouldn't halt on an overlap; it should record a structured, --json-inspectable, syncable conflict and hand it to a resolver. The sharpest tie-in: optimistic lift (§7.7) today must stop cleanly on a true overlap because there is nowhere to put the conflict; with conflict objects it could record the overlap and continue — exactly the "a fleet stops serializing through a human" goal §7.7 chases. This strengthens something Forklift already claims rather than chasing something jj owns. Sequence after the founding-bet items, but flag it as the natural next step on the agent track.

Done 2026-07-11 9.4b Native large-file handling

Forklift's streaming + bounded-memory + parallel-Blake3 design already positions it better than Git-without-LFS. Making large blobs a first-class chunked/streamed citizen (rather than Git's bolt-on git-lfs) turns an existing architectural strength into a headline — "native what Git needs an add-on for" — and removes a real adoption blocker for anyone with binaries or datasets. Extends the bounded-memory principle (§2) rather than bolting on. Open questions: chunking scheme, dedup granularity, and how the presigned byte plane (§4) streams a multi-GB object.

The design round (2026-07-11, adversarially reviewed) settled two new object types and the tree-level signal the review forced. A chunked file is a Recipe (ObjectType code 4 — an ordered list of chunk hashes/sizes plus a whole-file content hash) over Chunk leaves (code 5, capped at 4 MiB and enforced on both store and read); the recipe is an ordinary content-addressed object, so it dedups, packs, and delta-compresses like a blob for free. The adversarial catch: the review found that a chunk-unaware gc marks a file entry live by hash alone, never loading the object it names — so under the original tree-untouched representation (chunking signaled only inside the Recipe object) an already-shipped old binary's gc marks the recipe hash live, never descends into it, and silently sweeps every loose, unreferenced chunk as garbage: a chunked file becomes permanently unmaterializable, and gc reports success. This refuted the tree-untouched representation the design started from and forced two new DirEntryType codes, NormalChunked (5) and ExecutableChunked (6): an old binary's tree parser now fails loudly on the unrecognized code (the already-deployed from_code Err path, not new code an old binary would need to have), and gc's existing "error ⇒ delete nothing" behavior turns the silent loss into a loud no-op — continuing the standing pattern that every stage of this design work has had its blocker come from the same boundary/type-flip attack family §7.6 kept finding.

Chunking is content-defined and vendored, not delegated to a crate. A FastCDC-style boundary finder with frozen parameters (CHUNK_THRESHOLD_BYTES 8 MiB — the blob/recipe cutoff; CDC minimum/average/maximum 256 KiB/1 MiB/4 MiB) is vendored inline rather than pulled from the fastcdc crate, whose majors have changed both the gear table and the boundary algorithm — a routine dependency bump would otherwise silently relocate chunk boundaries and fork the signed tree hash between clients on different crate versions. The gear table and algorithm are pinned by golden test vectors tied to RECIPE_FORMAT_V1. MAX_OBJECT_BYTES (64 MiB) becomes the ceiling on the largest whole object any store or bundle accepts — a big tree or recipe, never a blob or chunk — with documented v1 hard limits (~762,000 entries in one directory, ~964 GiB for a flat chunked file; hierarchical recipes deferred). The unconditional defense for §5.0 D/7 is streaming: a new store_object_stream hashes and zstd-encodes an 'O'/'D' bundle-import payload incrementally to a temp file rather than a growing Vec, bounding memory regardless of declared length; the 64 MiB ceiling is policy layered on top of that streaming defense — refused before a byte is read, under a bumped bundle version — not the defense itself. compact --rechunk migrates grandfathered pre-chunking whole blobs over the ceiling into recipe+chunks, and ships with the same stage as the ceiling.

The remote plane rides the existing presigned byte plane, with one additive handshake field. Chunks and the recipe negotiate through the existing upload-targets/presigned-PUT/commit path — per-chunk dedup and resumability for free, no new endpoint shape — gated by an additive chunking capability field on the GET /v1/warehouse handshake (B1's server half: a chunk-aware client refuses to lift a chunked file to a remote that doesn't advertise it, rather than stranding chunks on a server whose gc would eat them). Verifying the AWS transport limits (API Gateway 10 MB, Lambda's 6 MB sync / 200 MB streamed) at the maintainer's request found upload-targets's existing 1,000-hash batching already safe, and one real gap: commit_lift's hash lists have no size bound today, closed by client-side pagination plus an additive more flag that decouples the server's staging-prefix sweep from per-batch verification. The commit-gate closure audit — on both server and client — must descend a Recipe-resolving hash and presence-check every chunk inside it, not just the recipe itself: this is the actual availability guarantee that a signed ref never points at missing history, elevated from an audit nicety to a hard commit-time requirement. Maintenance keeps chunks loose, never packed, never delta'd (a presigned-GET target must stay individually addressable, and CDC already captured the dedup delta would chase); recipes are packed and delta-compressed like blobs. gc's reachability walk is corrected to dispatch on the tree entry's DirEntryType before any load — an ordinary entry stays a zero-load mark, a *Chunked entry descends the recipe, presence-tolerant like the existing subtree check.

All four stages have shipped (2026-07-11). Staged as four independently shippable, independently reviewed stages: representation + local pipeline + the B1 local fix (Stage 1); the ceiling + D/7 closure (Stage 2); the streaming remote plane + the B1 server-half fix + the commit-pagination fix (Stage 3); audit's re-verify + scope-prune's chunk descent + the commit-gate's base-tree prune, closed out by a real-S3 LocalStack verification pass (Stage 4, below). Stage 2's migration item is revised from the plan below, one of the four decisions the maintainer settled in the design round: the round settled compact --rechunk (migrating a grandfathered pre-chunking blob above the ceiling into recipe+chunks) as shipping with Stage 2. Implementation surfaced the identity contradiction no migration mechanism escapes: a giant blob's hash is pinned by the signed tree entries that name it; a recipe of the identical content hashes differently; so every migration is either a signed-history rewrite that forges authorship, or a no-op that fixes nothing user-visible. compact --rechunk is therefore dropped, not deferred — stated plainly as a revision of the settled decision, not left implicit. The settled posture instead: a grandfathered giant stays readable and checkout-able locally forever (the ceiling gates writes and imports, never reads), every transport path — bundle and lift alike — refuses it honestly at the source (oversized_transport_unsupported), and no migration command exists. What §9.4b's completion leaves deferred, each honestly named, not implied: git-LFS interop (demand-gated — no user has asked for git-LFS-pointer round-tripping yet); the self-host chunk-pack-with-range-index (loose chunks face real inode pressure at 10M-chunk/multi-TB scale, noted in CHUNK_OBJECT_FORMAT.md, S3 unaffected since object storage has no inode limit); the code-side staging-session age sweep (an abandoned or endlessly-paginated lift's staged objects are never revisited by any code path — the S3 lifecycle rule on staging/ stays the documented operator MUST, not a code-side fix); and direct-head single-object streaming (a recorded STOP, not an omission — chunking already bounds those handlers to <= MAX_CHUNK_BYTES, and true streaming there is a store-read-API rework, not a §9.4b task). The full memo is retained outside the repo for implementation reference.

9.5 Ecosystem — the real gap is time, not code

The dominant practical gap versus Git (IDE plugins, CI integrations, hosting, Stack Overflow) is not a feature to implement — it is adoption, which takes users and time. The right code investment is the enabler already shipped: the stable machine interface (--json, stable error/exit codes, forklift mcp, §7.4). Everything downstream — editor integrations, CI runners, bots — builds on that surface. Recorded here so it is not mistaken for a buildable checkbox: past the machine interface, closing the ecosystem gap is mostly evangelism and third-party integration, not core work — with one exception, below.

The one piece worth building in-house, and soon (recorded 2026-07-08): the seed editor plugins. "Third parties will build the integrations" has a bootstrap problem — nobody writes a plugin for a VCS with no users, and the human half of the audience lives in an editor (agents drive the CLI/MCP; people review, stage and resolve conflicts in VS Code or IntelliJ). So the first VS Code and IntelliJ plugins are a deliberate Forklift deliverable, each in its own repo (like the hosting service), Apache/MIT like the rest. They are not core work and never touch forklift-core — they consume the §7.4 machine interface. Priority order: VS Code first, then IntelliJ.

Plan VS Code — fork the built-in Git extension, swap the backend

VS Code exposes a first-class SCM API (vscode.scm — source-control groups, gutter decorations, the diff view, the commit box), and the built-in Git extension is its reference implementation, MIT-licensed inside microsoft/vscode — so forking it and replacing the git backend is licence-clean and MIT-compatible with Forklift. The one correction to "just replace git calls with forklift calls": the backend must speak the stable machine interface (forklift --json, stable error/exit codes, conflicts --json — all shipped §7.4), never scrape human output, or the UI breaks the first time a message is reworded. The FORK-25 git aliases already give a git-shaped porcelain that maps onto the SCM UI's built-in vocabulary, so the MVP presents pallets→branches, stack→commit, consolidate→merge under the affordances the editor already draws — a small mapping round, not a blocker. Bonus: writing this plugin is the best forcing function we have to stress-test the machine interface and surface its gaps while there are still no users to break — reason enough to spike it early.

Plan IntelliJ IDEA — second, heavier

JetBrains exposes a VCS integration API (implement AbstractVcs plus the change-provider / history / diff providers); the reference is git4idea, Apache-2.0 in the IntelliJ Community repo — again licence-clean against Forklift's Apache/MIT. Registering a whole new VCS is a heavier lift than VS Code's source-control provider, so it follows VS Code once the machine-interface-as-editor-contract has been validated by the first plugin.

Timing. This is an adoption investment and it is all pre-announce, so it sits behind the one non-negotiable before announcing: that the core is trustworthy (Hardening milestone A — a store that can silently serve wrong bytes is not announceable). Net: the alias ships any time (a one-afternoon symlink), and the VS Code plugin starts as Hardening A closes and runs parallel to milestones B/C — it is a separate repo depending only on the shipped machine interface, not on the AWS head. It must land before the public announcement; a git competitor humans can't drive from their editor strands half the audience. Tracked as §5.0 milestone F.

9.6 History editing & rebase — a hard no and a scoped yes (revised 2026-07-06)

Correction of an earlier stance. A first draft of this section declined rebase wholesale as "fits badly." That conflated two operations that land on opposite sides of the philosophy. The honest split: rewriting published history is a permanent no; rewriting local, unpublished history fits fine and is already half-shipped.

Hard no Rewriting published history (force-push)

The remote is fast-forward-only CAS, no force-push (§4.2) — the single mutable-state consistency point the whole serverless design leans on. Rebasing a parcel that has already been lifted asks the remote to accept a non-fast-forward ref move. That is deliberately impossible and should stay so. "Rebase history others already have" is a real, permanent no — and it is load-bearing, not an oversight.

Fits — scoped Rewriting local, unpublished history

The append-only, one-way-door property is a fact about the office/trust chain, not about a working pallet: pallets are mutable pointers, undo already soft-resets a head, and nothing about a feature pallet's parcel sequence is sacred. The signing model does not forbid the rewrite either — a rewritten parcel is a new object, signed by the rewriter (honest: they produced it), and the parcel's Author/Stack action split (the same split import-git/export-git round-trip) preserves the original author while recording the rewriter as stacker. The model already has the slot.

The on-brand upgrade: git's "don't rebase public history" is a social convention people violate constantly. Forklift can make it enforced — a history-edit verb that refuses to touch any parcel reachable from a known remote head, and re-signs the ones it does rewrite. A footgun becomes a guarded primitive; that is exactly the kind of structured-safety move Forklift should prefer over git's honor system.

What you actually reach for rebase for, and Forklift's answer:

  • Cherry-pick — Done (§9.1 #8, shipped 2026-07-06). "Apply a parcel's diff onto the current pallet as a new, author-preserving, freshly-signed parcel" only adds — no rewrite, no force-push, no philosophical objection. It cannot match Pijul's duplicate-free elegance (snapshot systems never do), but a plain apply-a-diff cherry-pick is serviceable. Moved out of "declined."   • Guarded local replay — buildable, lower priority. A rebase-equivalent scoped to unpublished-or-new parcels, re-signing as it goes, enforcing "never rewrite what has been lifted" as code. Gate on real demand: deliver already covers the common "make it clean before sharing" case, so this earns its complexity only if users genuinely want curated multi-parcel PRs (reorder / reword / split).
  • Still declined: git-style interactive rebase over published history, and force-push. Not missing features — deliberately refused, and the refusal is load-bearing.

The deliver + first-class-conflicts + optimistic-lift trio dissolves most of the why behind rebase; a cherry-pick verb closes the sharpest remaining gap.

9.8 Git mirroring — continuous export/import as an adoption bridge (recorded 2026-07-08) Exploratory — needs design

The idea. A continuous mirror to and/or from a git remote — the way a self-hosted GitLab push-mirrors to GitHub — as the ongoing version of the one-time import-git/export-git already shipped (§7.8). The enabling fact: export-git is already fully deterministic — the same parcel produces the same git commit hash on every run (author/committer timestamps come from the parcel's stored actions, never now; identity from the on-chain operator id; tree order is sorted; dates are pinned via GIT_AUTHOR_DATE/GIT_COMMITTER_DATE). So idempotency falls out of content-addressing — a mirror needs no persisted parcel↔commit mapping table, which is normally the hard part.

Forklift → git (export mirror) — the easy, high-value direction. A v1 is nearly free: re-export to a scratch repo and git push; because export is deterministic, git dedupes by hash and the push fast-forwards. Pallets already map to refs/heads/<pallet> and consolidations already become real multi-parent git merge commits, so branches and merges survive. What is missing is small: relax export-git's fresh-dir-only guard (or export-to-scratch-then-push), plus a trigger — the provider-side events hook already fires EVENT_PALLET_UPDATED with old_head/new_head on every push (§8.13), exactly the ref-move signal a push-mirror hangs off, though a local mirror would want the client-side post-lift hook that §9.7 already flags as missing. It is lossy but acceptably so: the git mirror drops the office, @manifest provenance, per-parcel signatures and tags — a consumption shadow, not the authority — and is one-way (pushing directly to the mirror diverges it, the standard mirror rule).

git → Forklift (import mirror) — worth doing, more work, one honest caveat. It is blocked today by import-git's guards (fresh-warehouse only; refused once trust or pallets exist) and its within-a-single-run memoization. Continuous import needs new code: relax the guards, persist the git-hash→parcel map across runs, and incrementally fast-forward the mirrored pallets. The caveat is the signing story: git commits are not Forklift-signed, so mirrored parcels are either unsigned (the legacy boundary) or signed by the mirror bot — a service-class identity (§7.1) attesting "I imported git commit <sha>". Either way you get Forklift's structure and tooling (history, blame, the agent overlay) over git content, but not authentic author provenance. It is an ingestion/overlay, not a trust upgrade — which must be labelled honestly.

Prior art and the framing. Every serious git alternative ships a git bridge — jj uses git as its storage backend (the deepest form), Sapling interoperates with git, and hg-git / git-svn / git-p4 / Pijul all bridge. Mirroring is the lightest form, which makes a Forklift→git bridge an adoption necessity, not a nice-to-have. The framing that sells it: Forklift is your working VCS; git is your compatibility surface — you commit locally with signing and the agent-native features, and a live GitHub mirror keeps CI, code review, dependabot and not-yet-switched teammates working unchanged. On-thesis with the "reversible experiment, not lock-in" story (§7.8): not "you can leave" but "you can use both at once."

Recommendation. Export mirror first — low effort (determinism is already done), high adoption payoff, purely additive. Import mirror second, demand-gated, shipped with honest signing labels. Neither is on the §5.0 queue yet — record it as optionality, not commitment. One fidelity caveat for any round-trip claim: the original commit's timezone offset is dropped on import and re-emitted as +0000, so a git→Forklift→git round-trip changes hashes for non-UTC commits (harmless for a one-way mirror). Business / adoption implications (does seamless git mirroring reduce the pull to adopt or pay for Forklift hosting?) are out of scope for this design doc and recorded in the private planning notes.

2026-07-09PR #5 review — the delta "decompression-bomb guard" was not a guard. An automated review of the milestone-A/B PR caught two related things in the hardening work, and both were right. (1) delta_utils::decompress_delta was changed (T2) to stream its output instead of pre-allocating the declared decompressed length, which killed the cheap DoS — an 8-byte lie of u64::MAX no longer causes a capacity-overflow panic or an allocator abort. But the declared length was then used as the bound on the streamed read, and that length is attacker-controlled: with declen = u64::MAX the take() bounded nothing, so a small zstd frame could expand until memory ran out. The code, its comment, and BUNDLE_FORMAT.md all called it "a decompression-bomb guard"; it was not one. The fix uses an invariant already in the codebase rather than inventing a number: pack_utils has always refused to delta an object above 16 MiB ("deltating huge blobs costs more RAM/CPU than it saves"). That constant is now delta_utils::MAX_DELTA_TARGET_BYTES, shared by both writers — bundle_utils was silently deltating blobs of any size, which is precisely why no ceiling could be enforced on the read side — and decompress_delta now refuses a declared length above it before reading a byte of the frame, and requires the frame to reconstruct to exactly the declared length rather than merely not exceed it. So the bound is real and its soundness is a property of the format, not of the sender's honesty. (2) The regression test pinning the original fix asserted guarded.is_ok() || guarded.is_err() — tautologically true, catching nothing, next to a doc comment claiming the case "errors cleanly" when it in fact returned Ok. It now asserts the ceiling rejects u64::MAX and one byte over the limit, that a lie under the ceiling is caught by the exact-length check without allocating the 16 MiB it declared, and that an overrunning frame is rejected rather than truncated. Both new checks are mutation-verified: deleting the ceiling, and relaxing the exact-length check to "at most", each fail it. Recorded, not fixed (§5.0 D/7): the full-object 'O' record path on bundle import has the same shape — its declared length is the only bound and objects have no size ceiling — so a zstd bomb in a bundle from an untrusted remote can still expand unbounded. It is strictly better than the pre-allocation it replaced, since an attacker must now supply real compressed bytes rather than an 8-byte lie, but closing it needs a max-object-size policy entangled with §9.4b. The lesson is the standing rule again, one level up: a claimed invariant ships with its enforcement or its test — and a test that cannot fail is not a test.
